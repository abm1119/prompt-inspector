Prompt Inspector Technical Documentation

Prompt Inspector is a local analysis workbench for decomposing, scoring, stress testing, and rewriting prompts. It treats a prompt as an operational artifact: something that should be auditable, reproducible, safe under adversarial pressure, and clear enough for another human to maintain.

01Operating Principles

The application is designed around evidence rather than vibes. Each report separates structure, quality metrics, vulnerabilities, stress-test outcomes, and optimized rewrites so the user can understand why a prompt is strong or weak.

  • Inspectable structure: role, task, context, constraints, assumptions, and output format are surfaced before recommendations are made.
  • Comparable improvements: original and hardened scores are shown side by side so improvement claims can be challenged.
  • Adversarial readiness: stress results document whether common prompt attacks expose weak boundaries.
  • Portable artifacts: each run can produce local exports that preserve the original prompt, analysis, rewrite, and manifest.
The core product question is: can this prompt survive real use, another user's interpretation, and an adversarial attempt to bend its instructions?

02System Architecture

Prompt Inspector uses a small local web stack: FastAPI exposes the API, vanilla JavaScript renders the workbench, and the analysis modules in src/ perform parsing, improvement, stress testing, optimization, and exporting.

Browser

Collects the source prompt, renders tabs, and presents reports without a frontend build step.

FastAPI

Accepts inspection requests and coordinates the local pipeline through /api/inspect.

Analyzer

Extracts prompt type, role, task, assumptions, constraints, risks, and improved prompt fields.

Evaluator

Scores original and hardened prompts, records deltas, and prepares stress-test scenarios.

Exporter

Writes local artifacts so reports can be reviewed, shared, or compared later.

03Analysis Pipeline

A run begins when the user enters a prompt and selects Execute Analysis. The interface posts the prompt to the API, hides stale results, shows pipeline progress, and renders the returned report into focused tabs.

  1. Input validation: empty prompts are rejected before work begins.
  2. Semantic parsing: the prompt is decomposed into role, task, context, constraints, assumptions, and output expectations.
  3. Improvement cycle: the source prompt is rewritten into a safer, clearer, and more reusable version.
  4. Stress evaluation: adversarial variants are tested against the hardened prompt.
  5. Optimization pass: the system produces a lean rewrite and a strategy log that explains the editing choices.

04Scoring Model

Scores are intended as comparative signals, not absolute truth. The most useful reading is the delta between original and hardened prompts, paired with the structural explanation in the overview tab.

DimensionWhat It ChecksGood Prompt Signal
GrammarLinguistic correctness and readability.Instructions can be understood without repair.
ClarityWhether the request has one coherent interpretation.The model knows the task, scope, and success criteria.
SpecificityPresence of concrete constraints, examples, and boundaries.Ambiguous phrasing is replaced by operational instructions.
StructureOrdering and separation of role, task, context, and format.The prompt is skimmable and easy to maintain.
SafetyResistance to harmful, conflicting, or policy-bypassing instructions.Instructions define what to refuse, preserve, or escalate.
ReusabilityWhether the prompt can be reused with new inputs.Variable inputs are separated from stable rules.

05Stress Testing

Stress testing measures whether a hardened prompt holds its boundaries when exposed to hostile or messy user input. Each test reports an attack vector, pass/fail status, risk level, and observation.

Pass The hardened prompt preserved its intended behavior under the tested variant.
Review A failed or high-risk result should be treated as a prompt design issue, not only as a model behavior issue.

Typical vectors

  • Instruction override attempts that ask the model to ignore prior rules.
  • Role confusion where the user tries to replace the assistant's assignment.
  • Format attacks that force output outside the requested schema.
  • Boundary probing that asks for missing assumptions or prohibited content.

06API Reference

POST /api/inspect

Runs the full inspection pipeline and returns the analysis, comparison, stress results, optimizer output, request identifier, and export path. 

fetch("/api/inspect", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ prompt: "Your prompt text..." })
});

POST /api/optimize

Runs only the optimization stage. Use this endpoint when the user needs a concise rewrite and improvement strategy without the full inspection report. 

{
  "request_id": "uuid",
  "result": {
    "optimized_prompt": "...",
    "steps": [
      { "name": "Remove ambiguity", "explanation": "..." }
    ]
  }
}

07Operations

The service is intentionally local and lightweight. Static assets are served from web/, API responses are generated in process, and HTML, CSS, and JavaScript are sent with no-cache headers for fast iteration.

ConcernOperational Note
StartupRun the FastAPI app with Uvicorn and open the local web root.
ExportsArtifacts are written under the local exports directory and referenced in the response.
Failure handlingThe workbench shows API errors in a visible toast and restores the run button after completion.
DocumentationThe documentation iframe is lazy-loaded only when the footer Documentation button is clicked.
